PROJECT OVERVIEW
----------------
Pratipal is a full-stack e-commerce and wellness service booking platform built
for "Pratipal Healing" (pratipal.in). It supports physical product sales
(candles, roll-ons, salts), online course enrollment, healing session bookings,
dynamic landing pages, a blog, and a full admin dashboard.

The platform was originally built on Supabase and has been fully migrated to
MongoDB Atlas. Image storage uses Cloudflare R2 (with AWS S3 as fallback).

  - TypeScript 5.7.3
  - Node.js (runtime)

DATABASE
  - MongoDB 7.1.0 (primary database)

  - Sonner (toast notifications)
  - class-variance-authority + clsx + tailwind-merge (utility styling)
  - tailwindcss-animate (animation utilities)

  - jsonwebtoken 9.0.3 (JWT signing/verification)
  - bcryptjs 3.0.3 (password hashing)
  - Cookie-based sessions (HttpOnly, Secure in prod, SameSite=Lax)
  - 30-day token expiry

PAYMENT PROCESSING
  - Razorpay 2.9.6 (Indian payment gateway)

SHIPPING & LOGISTICS
  - Shiprocket API v2 (order creation, tracking, cancellation)

STATE MANAGEMENT

DRAG & DROP

UTILITIES

  - dotenv 17.3.1 (env loading in scripts)

ANALYTICS / REVIEWS

8. LIBRARY MODULES (src/lib/)
================================================================================

auth.ts
  - JWT signing (signToken), verification (verifyToken)
  - getUserFromRequest: extracts user from cookie in API routes
  - getSessionCookieOptions: returns secure cookie config
  - Cookie name: "pratipal_session", 30-day expiry
  - Interfaces: AuthUser, JwtPayload, AuthenticatedUser

mongodb.ts
  - Singleton MongoDB connection with global cache
  - connectDB(): returns cached mongoose connection
  - Pool: maxPoolSize=10, serverSelectionTimeout=5s, socketTimeout=45s
  - Logs connection attempts and errors

db.ts
  - getDB(): connects to MongoDB and returns all model references
  - Exports: AuthUser, Customer, CustomerAddress, Product, Category,
    Order, OrderItem, CartItem, LandingPage, Media, InvitationRequest,
    HeroSection, ShippingSettings, SessionBooking, Service, Course, Blog

r2-client.ts (R2Storage class)
  - uploadFile(file, folder): uploads File object to R2
  - uploadBuffer(buffer, fileName, contentType, folder): uploads Buffer
  - deleteFile(key): removes file from R2
  - extractKeyFromUrl(url): parses R2 key from public URL
  - isConfigured(): validates all R2 env vars are present
  - Uses @aws-sdk/client-s3 with R2 endpoint
  - 10MB file size limit, images only
  - ACL: public-read, CacheControl: 1 year

9. AUTHENTICATION FLOWS
================================================================================

--- ADMIN AUTHENTICATION ---

1. Admin visits /admin/login
2. Submits email + password to POST /api/auth/login
3. Server: finds AuthUser by email, bcrypt.compare(password, hash)
4. On success: signToken(user) → JWT with {sub, email, role, full_name}
5. Sets HttpOnly cookie "pratipal_session" (30 days)
6. Admin pages call GET /api/auth/me → getUserFromRequest(req)
7. JWT verified → returns AuthenticatedUser
8. Logout: POST /api/auth/logout → clears cookie

--- CUSTOMER AUTHENTICATION ---

1. Customer registers via POST /api/auth/register
   - Validates email uniqueness
   - bcrypt.hash(password, 10)
   - Creates Customer document
   - Optionally sends verification email
2. Customer logs in via POST /api/auth/customer-login
   - Finds Customer by email
   - bcrypt.compare(password, hash)
   - Signs JWT → sets customer session cookie
3. CustomerAuthProvider checks auth on mount via GET /api/auth/customer-me
4. Customer state available via useCustomerAuth() hook
5. Logout: POST /api/auth/customer-logout → clears cookie

--- PASSWORD RESET ---

1. POST /api/auth/forgot-password with email
2. Generates reset_token (UUID) + reset_token_expires (1 hour)
3. Saves to Customer document
4. Sends email with reset link
5. Customer clicks link → submits new password with token
6. Server validates token + expiry → updates password_hash → clears token

================================================================================
10. PAYMENT FLOW (Razorpay)
================================================================================

--- PRODUCT ORDER PAYMENT ---

1. Customer adds items to cart, proceeds to checkout
2. Checkout page collects shipping/billing address
3. POST /api/razorpay/create-order
   - Creates Razorpay order with amount (paise)
   - Returns {razorpay_order_id, amount, currency, key_id}
4. Frontend opens Razorpay checkout modal
5. Customer completes payment
6. POST /api/razorpay/verify-payment
   - Verifies HMAC-SHA256 signature: razorpay_order_id + "|" + razorpay_payment_id
   - On success: creates Order document, updates payment_status="paid"
   - Triggers Shiprocket order creation
   - Clears cart
7. Redirect to /order-confirmation

--- SESSION/COURSE BOOKING PAYMENT ---

1. Customer selects service/course on booking page
2. Fills in personal details (name, email, phone, WhatsApp)
3. POST /api/sessions/create-payment
   - Creates SessionBooking with payment_status="pending"
   - Creates Razorpay order
   - Returns booking_number + Razorpay order details
4. Frontend opens Razorpay modal
5. POST /api/sessions/verify-payment
   - Verifies signature
   - Updates SessionBooking: payment_status="paid", booking_status="confirmed"
   - Sends confirmation email to customer
   - Sends notification email to admin
6. Redirect to /booking-success

11. SHIPPING FLOW (Shiprocket)
================================================================================

1. Order is created with payment_status="paid"
2. Admin reviews order in dashboard
3. Admin triggers "Create Shipment" → PUT /api/admin/orders/[id]
4. Server calls createShiprocketOrder(payload):
   - Maps order data to Shiprocket format
   - Sends to Shiprocket API
   - Stores shiprocket_order_id + shiprocket_shipment_id on Order
5. Shiprocket assigns AWB (tracking number)
6. Order status updated to "processing" → "shipped"
7. Customer can track via GET /api/shipping/track?shipment_id=...
   - Calls getShiprocketTracking(shipmentId)
   - Maps status via mapShiprocketStatus()
   - Updates tracking_history on Order
8. Status progression:
   order_received → processing → packed → shipped → out_for_delivery → delivered

12. FILE UPLOAD FLOW (Cloudflare R2)
================================================================================

1. Admin selects file in UploadZone component
2. POST /api/upload/[type] with multipart/form-data
3. Server: R2Storage.isConfigured() check
4. R2Storage.uploadFile(file, folder):
   - Validates: image only, max 10MB
   - Generates UUID filename
   - Uploads to R2 bucket via PutObjectCommand
   - ACL: public-read, CacheControl: 1 year
5. Returns {url, key, fileName}
6. URL stored in model (product.featured_image, course.featured_image, etc.)
7. Fallback: if R2 not configured, saves to /public/uploads/ locally

Folder structure in R2:
  uploads/     → general uploads
  products/    → product images
  courses/     → course images
  services/    → service images

18. SECURITY CONSIDERATIONS
================================================================================

- Passwords hashed with bcryptjs (cost factor 10)
- JWT tokens stored in HttpOnly cookies (not accessible via JS)
- Secure flag enabled in production
- SameSite=Lax prevents CSRF on cross-site requests
- API routes validate authentication via getUserFromRequest()
- Admin routes require role="admin" in JWT payload
- File uploads restricted to images only, 10MB max
- CORS configured for API routes (currently open — restrict in production)
- Razorpay payment verification uses HMAC-SHA256 signature validation
- Shiprocket token cached server-side (not exposed to client)
- Environment variables never exposed to client (except NEXT_PUBLIC_ prefix)
